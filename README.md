HabitTracker

Настольное приложение для контроля и формирования полезных привычек, разработанное в рамках курсового проекта по дисциплине «Проектная деятельность» на кафедре «Комплексная защита информации» ОмГТУ.

Приложение функционирует в полностью автономном офлайн-режиме, обеспечивая абсолютную конфиденциальность и безопасность персональных данных пользователя за счет локальной изоляции информации.
🚀 Ключевые возможности

    Управление (CRUD): Создание, просмотр, редактирование параметров и удаление привычек (название, описание, тип расписания, целевые показатели, дата старта).

    Гибкие режимы расписания: Поддержка ежедневных циклов (daily) и гибких еженедельных планов (weekly, например, определенное число дней в неделю).

    Глобальное целеполагание: Установка и автоматическое отслеживание прогресса достижения долгосрочной цели общего количества выполнений по каждой привычке (total_target).

    Удобное логирование: Ежедневная фиксация или отмена фактов выполнения за любую выбранную календарную дату с автоматической защитой от дублирования записей.

    Аналитическое ядро: Автоматический расчет параметров регулярности — текущих и максимальных (исторический рекорд) серий непрерывного выполнения (streaks) на основе стандартов ISO-календаря.

    Наглядная визуализация: Отображение динамических прогресс-баров выполнения глобальной цели и интерактивной календарной сетки активности за скользящие периоды времени (7 и 30 суток).

    Локальная безопасность: Полное отсутствие сетевых запросов и внешних API; данные изолированы внутри локального файла СУБД SQLite.

📂 Структура проекта

Архитектура программного продукта построена по модульному принципу со слабой связанностью слоев представления, бизнес-логики и хранения данных:

    main.py — главная точка входа, координация абсолютных путей, инициализация менеджеров слоев и запуск основного цикла событий графического интерфейса.

    database.py — низкоуровневый менеджер СУБД SQLite (DatabaseManager), контекнтный менеджер безопасных транзакций, инициализация физической схемы таблиц, индексов и механизмов автоматической миграции.

    services.py — слой бизнес-логики приложения, валидация пользовательского ввода, реализация алгоритмов расчета серий непрерывности и агрегации статистических показателей.

    models.py — декларация строго типизированных структур обмена данными между слоями приложения на базе стандартных дата-классов (@dataclass).

    ui.py — слой графического интерфейса пользователя (GUI) на базе библиотеки Tkinter (модуль ttk), кастомные виджеты адаптивного скроллинга и модальные окна диалогов.

    tests/ — каталог с автоматизированными модульными тестами для независимой проверки корректности выполнения вычислительных алгоритмов сервисного слоя.

💾 Схема базы данных

Физическая структура реляционных данных развернута локально и состоит из двух таблиц, связанных отношением «один-ко-многим» с директивой каскадного удаления:

    habits — мастер-таблица метаданных привычек:

        id (INTEGER PRIMARY KEY AUTOINCREMENT) — уникальный идентификатор;

        title (TEXT NOT NULL) — название習慣;

        description (TEXT DEFAULT '') — краткое описание или цель;

        schedule_type (TEXT NOT NULL) — тип расписания ('daily' / 'weekly');

        schedule_value (INTEGER NOT NULL) — плановое количество выполнений;

        total_target (INTEGER NOT NULL) — долгосрочная цель общего числа выполнений;

        start_date (TEXT NOT NULL) — дата начала отслеживания (YYYY-MM-DD);

        created_at / updated_at (TEXT) — системные метки времени записи и модификации.

    habit_logs — журнал хронологии отметок выполнения:

        id (INTEGER PRIMARY KEY AUTOINCREMENT) — уникальный идентификатор лога;

        habit_id (INTEGER NOT NULL, FK -> habits.id ON DELETE CASCADE) — ссылка на мастер-привычку;

        log_date (TEXT NOT NULL) — целевая дата отметки выполнения;

        status (INTEGER NOT NULL DEFAULT 1) — маркер выполнения (0 или 1);

        comment (TEXT DEFAULT '') — необязательный текстовый комментарий к отметке;

        created_at (TEXT) — метка времени создания записи лога.

Для оптимизации скорости выполнения аналитических выборок по истории дат инициализируется композитный индекс idx_logs_habit_date.

🛠 Технологический стек

Программный продукт спроектирован с использованием стандартных средств языка Python и не требует установки сторонних внешних библиотек (privacy by design):

    Язык разработки: Python 3.11+

    Графическая оболочка: Tkinter (модуль ttk)

    Управление данными: СУБД SQLite (встроенная библиотека sqlite3)

    Работа со временем: Стандартные системные модули datetime и calendar
